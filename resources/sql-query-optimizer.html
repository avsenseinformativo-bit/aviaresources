Why is my query slow?

Database performance is often the bottleneck of modern applications. A single bad query can bring down a server. Optimization isn't magic; it's about understanding how the database engine reads data.

1. The "SELECT *" Anti-Pattern

Never select columns you don't need. It increases network traffic, memory usage, and prevents "Index Only Scans".

‚ùå Bad: SELECT * FROM users
Fetches generic columns like 'password_hash' or huge 'bio' text fields unnecessarily.

‚úÖ Good: SELECT id, email, name FROM users
Fetches only what is needed. Allows the DB to potentially read directly from the index.

2. The Power of Indexing (B-Tree)

Imagine finding a name in a phone book. If it's sorted (Indexed), it takes seconds (Logarithmic time). If it's random (Heap), you have to read every page (Linear time).

Rule of Thumb: Index columns that are frequently used in WHERE, JOIN, and ORDER BY clauses.

3. The "LIKE" Trap

Using LIKE '%term%' forces a full table scan because the wildcard is at the start. The database cannot use a standard B-Tree index.

‚ö†Ô∏è Performance Warning
LIKE '%paid%' prevents index usage.
Fix: Use Full Text Search (Postgres tsvector) or ElasticSearch for search functionality.

4. N+1 Query Problem

This happens when your code executes a query for every row in a result set. Example: Fetching 100 users, then running 100 separate queries to get their latest order. Fix: Use JOIN or eager loading to fetch everything in 1 or 2 queries.
